

# Πλάνο: Χειροκίνητη Προσθήκη Κρατήσεων/Εισιτηρίων από τον Επιχειρηματία

## Τι Κάνει
Ο επιχειρηματίας μπορεί να πατήσει ένα κουμπί **"+"** μέσα στη Διαχείριση και να προσθέσει χειροκίνητα ένα άτομο/κράτηση. Η εγγραφή εμφανίζεται κανονικά στον πίνακα σε αλφαβητική σειρά. Η κατάσταση ξεκινάει ως **"—"** και ο επιχειρηματίας μπορεί να την αλλάξει σε **"Ήρθε"** ή **"Δεν ήρθε"**.

## Κατάσταση (Status) για Χειροκίνητες Εγγραφές

- **Αρχική κατάσταση:** Εμφανίζεται **"—"** (παύλα)
- **Tappable:** Ο επιχειρηματίας πατάει την παύλα → εμφανίζεται μικρό dropdown:
  - **"—"** (δεν ξέρουμε ακόμα)
  - **"Ήρθε"** → εμφανίζεται πράσινο check-in badge
  - **"Δεν ήρθε"** → εμφανίζεται no-show
- Δεν υπάρχει QR code ή confirmation code

## CRM — Πώς θα Εμφανίζονται

Αυτό είναι κρίσιμο: ο επιχειρηματίας μπορεί να γράψει ένα όνομα αλλά η κράτηση να είναι για 10 άτομα.

### Πώς το κάνουν οι μεγάλες πλατφόρμες:

**Προσέγγιση A — "Booker-Centric" (SevenRooms, OpenTable)**
- Μόνο ο **booker** (αυτός που έκανε την κράτηση) μπαίνει στο CRM ως εγγραφή
- Το `party_size` καταγράφεται στο ιστορικό του booker (π.χ. "10 άτομα, 28 Μαρτίου")
- Τα υπόλοιπα 9 άτομα **δεν** δημιουργούν ξεχωριστές εγγραφές CRM
- **Πλεονεκτήματα:** Καθαρό CRM, χωρίς "φαντάσματα", εύκολο spend tracking
- **Μειονεκτήματα:** Δεν γνωρίζεις ποιοι ήταν οι υπόλοιποι 9

**Προσέγγιση B — "Ghost Profiles" (αυτό που ήδη κάνετε)**
- Ο booker + κάθε μέλος του party δημιουργούν ξεχωριστή εγγραφή
- Σε χειροκίνητη εισαγωγή όμως, ο επιχειρηματίας δεν γράφει 10 ονόματα — γράφει μόνο ένα

### Πρόταση για το FOMO:

Για τις **χειροκίνητες εγγραφές**, ακολουθούμε την **Booker-Centric** προσέγγιση:

1. Δημιουργείται **ένα ghost profile** στο CRM με το όνομα που έγραψε ο επιχειρηματίας
2. Το `party_size` αποθηκεύεται στην κράτηση (π.χ. 10 άτομα)
3. Στο CRM profile αυτού του ατόμου, φαίνεται:
   - **Επισκέψεις:** 1 (αν σημειωθεί "Ήρθε")
   - **Μέσος αριθμός ατόμων:** 10
   - **Έξοδα:** 0€ (εκτός αν ο επιχειρηματίας βάλει χειροκίνητα actual spend)
4. Αν ο επιχειρηματίας θέλει να καταγράψει τα ονόματα των υπόλοιπων, μπορεί να τα γράψει στο πεδίο **Σημειώσεις**

Αυτό σημαίνει: **1 χειροκίνητη εγγραφή = 1 CRM profile**, ανεξάρτητα από το party_size.

## Database Αλλαγές

1. **Νέα στήλη `is_manual_entry`** στον πίνακα `reservations` (boolean, default false)
2. **Νέα στήλη `is_manual_entry`** στον πίνακα `tickets` (boolean, default false)
3. **Νέα στήλη `manual_status`** στον πίνακα `reservations` — τιμές: `null`, `'arrived'`, `'no_show'` (μόνο για manual entries)
4. **RLS policy** για INSERT: ο ιδιοκτήτης της επιχείρησης μπορεί να προσθέσει εγγραφές στα δικά του events/business

## UI Αλλαγές

### Κουμπί "+"
- Τοποθετείται πάνω δεξιά στον πίνακα Διαχείρισης
- Εικονίδιο `Plus` (lucide), στρογγυλό, subtle styling (ίδιο χρώμα με τα tabs)
- Πατώντας το → ανοίγει ένα compact dialog με τα πεδία του αντίστοιχου τύπου

### Dialog Προσθήκης
- Τίτλος: "Προσθήκη" / "Add"
- Πεδία ανάλογα με τον τύπο (βλ. πίνακα παραπάνω)
- Κουμπί "Αποθήκευση" → εισαγωγή στη βάση → refresh πίνακα
- Η νέα εγγραφή εμφανίζεται αλφαβητικά

### Κατάσταση στον Πίνακα
- Αντί για badge, εμφανίζεται **"—"** ως clickable text
- Click → μικρό popover με 3 επιλογές: "—", "Ήρθε", "Δεν ήρθε"
